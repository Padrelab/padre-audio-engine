# DualSdWavLoopI2s Development Log

## Назначение

Этот документ фиксирует эволюцию примера `DualSdWavLoopI2s`, текущую архитектуру и цепочку найденных проблем. Он нужен как технический дневник, чтобы:

- быстро восстановить контекст спустя время;
- понимать, как сейчас устроен движок;
- не повторять уже пройденные гипотезы;
- видеть, какие проблемы уже решены, а какие еще остаются риском.

Текущая рабочая ветка прошивки на момент этого протокола:

- `Build: dual-sd-wav-i2s-rtos-q49152-r3`

  - `MPR121_SDA = 35`
  - `MPR121_SCL = 36`

## Что делает система сейчас

Система читает WAV-файлы с SD-карты из двух независимых плейлистов:

- `/music`
- `/foley`

Оба плейлиста играют одновременно и бесконечно зациклены. Каждый источник декодируется отдельно, затем оба потока микшируются в один stereo PCM поток и отправляются на I2S DAC.

Управление с MPR121:

- электрод `0` -> следующий трек `music`
- электрод `1` -> следующий трек `foley`
- электрод `2` -> громкость `-1`
- электрод `3` -> громкость `+1`

## Текущая архитектура

### 1. Источники и плейлисты

При старте система:

1. инициализирует SD;
2. сканирует `/music` и `/foley`;
3. оставляет только `.wav`;
4. проверяет WAV-заголовки;
5. выбирает общую частоту дискретизации;
6. отбрасывает WAV, у которых sample rate не совпадает с целевой.

Следствие:

- пример не занимается runtime resampling;
- все воспроизводимые WAV должны иметь одну и ту же частоту дискретизации.

### 2. Голоса

Используются два независимых экземпляра `LoopingWavVoice`:

- один для `music`
- один для `foley`

Каждый голос:

- держит собственный список треков;
- умеет открыть следующий WAV;
- декодирует WAV через `WavDecoder`;
- умеет принудительно перейти на следующий трек;
- поддерживает зацикливание плейлиста;
- ведет подробную статистику таймингов.

Начиная с `r3`, у каждого голоса есть собственный небольшой PCM reserve buffer:

- `kVoicePcmBufferSamples = 8192`
- refill идет небольшими чанками
- это снижает вероятность слышимого артефакта, если конкретный `source.read()` внезапно тормозит

### 3. Микширование

`VoiceMixer` берет данные у обоих голосов и суммирует их в общий `int16_t` буфер:

- `music gain = 0.60`
- `foley gain = 0.60`
- есть saturating clamp

### 4. Вывод в I2S

Схема вывода:

1. `VoiceMixer` формирует PCM блок;
2. блок записывается в software queue `BufferedI2sOutput`;
3. queue постепенно сливается в `Esp32StdI2sOutputIo`;
4. `Esp32StdI2sOutputIo` пишет в ESP-IDF `i2s_std` TX канал.

Текущие ключевые буферы:

- software PCM queue:
  - `kSinkQueueSamples = 49152`
- startup prebuffer:
  - `kStartupPrebufferSamples = 24576`
- рабочий target refill:
  - `kQueueRefillTargetSamples = 36864`
- I2S DMA:
  - `dma_desc_num = 32`
  - `dma_frame_num = 512`
  - `work_samples = 2048`

### 5. Управление и задачи

Начиная с RTOS-версии, аудио-обслуживание вынесено в отдельную FreeRTOS task:

- `audio_service`

Main loop теперь не делает тяжелую SD/I2S работу. Он:

- обслуживает touch;
- печатает `DIAG`;
- не вмешивается напрямую в состояние голосов.

Touch-команды больше не дергают голоса напрямую. Вместо этого:

1. touch складывает команды в `PendingControlState`;
2. audio task забирает эти команды;
3. именно audio task применяет:
   - `next music`
   - `next foley`
   - `volume +/-`

Это было сделано, чтобы убрать гонки между UI и аудио-потоком.

### 6. Переключение треков

`next` не исполняется мгновенно в обработчике касания. Вместо этого:

- запросы на `next` коалесцируются в окно `60 ms`;
- серия быстрых нажатий схлопывается в один реальный переход на несколько треков вперед;
- сам переход выполняется только когда queue уже достаточно заполнена.

Это убрало ситуацию, когда серия быстрых `next` сама же и выжигала весь аудио-буфер.

## Как читать DIAG

`DIAG event=queue-low` или `DIAG event=queue-empty` сейчас означает следующее:

- software PCM queue заметно просела;
- но это не всегда означает слышимую проблему.

Критичные поля:

- `queue trig/now/cap`
  - насколько просела software queue;
- `loop avg/max`
  - время одного цикла audio task;
- `svc avg/max`
  - время `serviceAudio()`;
- `budget`
  - сколько раз цикл уперся в budget;
- `pump avg/max`
  - время слива queue в I2S;
- `write short`
  - если это больше нуля, software queue не смогла принять весь PCM блок;
- `mix zero`
  - если больше нуля, микшер иногда не выдавал сэмплы;
- `read avg/max/last`
  - время чтения и декодирования на уровне конкретного голоса;
- `open last/max`
  - время `source.open`;
- `next/close/dec`
  - длительности перехода между треками;
- `short`
  - голос не смог выдать полный кусок PCM;
- `buf=last/peak`
  - состояние локального PCM-буфера конкретного голоса.

Практическое правило:

- `queue-low` без `short`, без `mix zero`, без `write short` и без слышимого артефакта считается near miss, а не реальным сбоем.

## Хронология проблем и решений

### Этап 1. Базовый пример

Сделан новый пример в `examples/DualSdWavLoopI2s`:

- SD -> WAV -> dual playlist -> mixer -> I2S
- два каталога `/music` и `/foley`
- бесконечный repeat

### Этап 2. Добавление MPR121

Добавлены сенсоры:

- `0` следующий `music`
- `1` следующий `foley`
- `2` громкость вниз
- `3` громкость вверх

### Этап 3. Первичная диагностика

Был добавлен периодический многословный `DIAG`.

Наблюдение:

- сами `Serial.printf` начали вызывать легкие слышимые обрывы;
- стало ясно, что диагностика мешает realtime пути.

Решение:

- периодический дамп убран из горячего пути;
- введен event-driven `DIAG`, который печатается только на `queue-low` / `queue-empty`.

### Этап 4. Поиск первого настоящего источника статтера

По первым логам стало видно:

- queue реально выгорала до нуля;
- `loop max` и `svc max` могли уходить в секунды;
- это было не steady-state ограничение I2S, а длинная блокировка main loop.

Подозреваемые тогда:

- тяжелый `DIAG`
- переключение трека в горячем пути

### Этап 5. Ограничение монополизации CPU

Были введены:

- budget на один проход `serviceAudio()`;
- уменьшенный размер одного refill chunk;
- более высокий refill target;
- hysteresis для `DIAG`.

Эффект:

- один проход `serviceAudio()` перестал слишком долго монополизировать CPU;
- но редкие SD stalls все еще могли приводить к near miss.

### Этап 6. Отложенное и коалесцированное `next`

По логам стало видно, что быстрые `next` сами по себе могут быть дорогими из-за:

- `close`
- `open`
- `decoder.begin`

Решение:

- `next` перестал исполняться сразу;
- запросы стали коалесцироваться;
- реальный switch стал выполняться только при безопасной заполненности queue.

Эффект:

- быстрые переключения стали заметно стабильнее;
- `next/open/close` перестали быть главным источником слышимого сбоя.

### Этап 7. Ограничение размера одного SD read

Логи показывали редкие `read max` порядка:

- `80 ms`
- `160+ ms`

Была добавлена граница на один `source.read()` внутри `WavDecoder`.

Эффект:

- decoder стал реже зависать на слишком большом чтении;
- но полностью длинные stalls SD-карты это не устранило.

### Этап 8. Увеличение queue и переход к RTOS

После этого стало ясно:

- однопоточная архитектура сама по себе уязвима;
- touch, serial-лог и audio service борются за один loop.

Решение:

- `serviceAudio()` вынесен в отдельную FreeRTOS task;
- touch переведен в command queue/state;
- `DIAG` переведен на snapshot модель.

### Этап 9. Watchdog regression

Первая RTOS-версия вызвала перезапуски:

- `audio_service` душил `IDLE0`
- срабатывал task watchdog

Причина:

- задача крутилась слишком агрессивно и почти не блокировалась

Решение:

- audio task стала гарантированно блокироваться хотя бы на 1-2 ms через `ulTaskNotifyTake`

Эффект:

- постоянные reboot исчезли

### Этап 10. Per-voice PCM buffering

После RTOS-перехода стало видно:

- общая queue уже часто жива;
- но редкие сильные искажения могли оставаться;
- это стало похоже на проблему конкретного голоса, а не всего sink.

Решение:

- у каждого `LoopingWavVoice` появился свой локальный PCM reserve buffer;
- в `DIAG voice=...` добавлены поля:
  - `short`
  - `buf`

Текущий эффект по последним логам:

- `queue-low` еще встречается;
- но `short=0`;
- `buf` у обоих голосов не падает в ноль;
- слышимых искажений в ряде случаев уже нет.

## Что уже решено

Решенные проблемы:

- базовое dual-WAV воспроизведение с SD и I2S;
- одновременный бесконечный repeat плейлистов `/music` и `/foley`;
- управление через MPR121;
- отделение диагностики от realtime пути;
- подавление статтеров от `Serial.printf`;
- устранение части статтеров от `next/open/close`;
- устранение watchdog reset RTOS-версии;
- улучшенная устойчивость к редким SD stalls;
- безопасная модель взаимодействия touch и audio task.

## Что остается риском

Даже сейчас логами подтверждено, что SD/FS иногда выдает очень длинные задержки чтения:

- порядка `70-170 ms`

Это не выглядит как чисто программная ошибка микшера или I2S. Скорее это комбинация:

- особенностей SD-карты;
- файловой системы;
- фрагментации/внутренних erase-пауз карты;
- блокирующего характера `FS::read`.

Система сейчас лучше их переносит, но фундаментально длинные stalls как явление никуда не делись.

## Как продолжать работу

Если снова появятся слышимые артефакты, собирать нужно:

1. boot-лог со строкой `Build: ...`
2. первый `DIAG` в момент проблемы

На что смотреть в первую очередь:

- `queue trig/now`
- `short`
- `buf`
- `write short`
- `mix zero`
- `read max/last`
- `open max`

Правила интерпретации:

- `queue-low` + `short=0` + `buf>0` + нет артефакта:
  - near miss, но система выдержала
- `queue-low` + `short>0` у одного голоса:
  - конкретный голос не успел выдать PCM
- `write short>0`:
  - проблема на уровне приема в sink queue
- `mix zero>0`:
  - микшер в какой-то момент не получил данных вовсе
- `queue-empty`:
  - это уже реальный underrun общей очереди

## Практические выводы по движку

### Что оказалось неверной гипотезой

- не всякий `queue-low` означает слышимый сбой;
- не всякий статтер был связан с `next`;
- не сам I2S write оказался главным узким местом.

### Что оказалось верной гипотезой

- длинные `Serial.printf` в горячем пути реально вредят аудио;
- длинные SD stalls существуют и хорошо видны в логах;
- разделение control-plane и audio-plane критично;
- локальная буферизация на уровне голоса полезна, а не только на уровне общего sink.

## Текущее состояние на момент протокола

По последним логам `r3`:

- RTOS-версия загружается корректно;
- watchdog reset исправлен;
- сенсоры работают;
- быстрые переключения больше не воспроизводят ранние ошибки;
- `queue-low` все еще может фиксироваться;
- но в последних известных логах при `queue-low` уже не было слышимых искажений;
- `short=0`, `buf` у голосов оставался ненулевым.

Текущая оценка:

- система находится в рабочем состоянии;
- основная критическая деградация устранена;
- оставшаяся зона наблюдения связана с редкими длинными stalls SD-карты и тем, насколько хорошо их переживает per-voice buffering.

## Что важно не забыть при следующем возвращении

- сначала сверять `Build:` в boot-логе;
- не сравнивать логи разных прошивок без этого;
- помнить, что старые логи с `queue cap=32768` относятся к более ранней архитектуре;
- не возвращать периодический подробный `DIAG` в горячий путь;
- не трогать `SD_CS`, он закреплен на `8`.
